Parakeet sidecar fallback chain

warm sidecar → crash-isolated helper → direct subprocess · all in-process in whichever binary has the feature

Parakeet has no in-process model (unlike whisper, which is linked in) — it runs the external parakeet.cpp binaries. So transcribe_with_parakeet tries three execution layers in order. Layer 1 is the only one that keeps the model warm; Layers 2 and 3 reload it per call. The whole chain only runs in a binary compiled with --features parakeet — otherwise the gate returns EngineNotAvailable before any of this.

The socket protocol. Newline-delimited JSON over a Unix domain socket (request_sidecar :1007). Request → {request_id, audio_path, decoder, timestamps, use_vad, beam_width, lm_path, lm_weight, boost_phrases, boost_score}. Response → {ok, request_id, text, elapsed_ms, word_timestamps, error}. Audio is passed by file path, not streamed. Request timeout = (audio_secs × 2).clamp(120s, 30min).
FP16 sticky-downgrade. On Apple Silicon the sidecar launches with --fp16, but MPSGraph crashes on certain shapes. classify_start_failure :1105 matches signatures like MPSGraph / requires the same element type, downgrades to fp32, and persists a fingerprint to parakeet-fp16-blacklist.json so every future launch for that model+binary+gpu skips fp16. Reset via parakeet_fp16_blacklist_reset.

Why "sidecar" only describes Layer 1. Layer 1's example-server is a long-lived companion process holding the model resident — that's the sidecar. Layers 2 and 3 are spawn-per-utterance and reload the model each call, so they're fine as fallbacks but unviable for live transcription. (Note: "sidecar" is overloaded in this repo — it also means companion files like live-transcript-status.json, and the recording-sidecar live-transcript mode. Those are unrelated to this process.)

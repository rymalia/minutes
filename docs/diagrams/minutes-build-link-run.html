minutes · verified from source @ 0.18.14

build → link → run

one library crate, two binaries that embed it, one TypeScript server that shells out

minutes-core is a library crate — it has no main() and compiles to a .rlib, so it cannot run on its own. minutes-cli and minutes-app are binary crates that depend on it via a local path dependency. At link time the compiler copies core's compiled code into each binary (static linking) — so each binary carries its own embedded copy. The MCP server is the asterisk: it is TypeScript, embeds nothing, and reaches core by running the minutes CLI.

Two axes of "can they be out of sync?"

Within one checkout → No. Both binaries use a path dependency on the same crates/core/ folder, and core inherits the one workspace version. At a given commit there is exactly one core source, so cli and app embed an identical core. They cannot diverge from the same tree.
Across install moments → Yes. The installed CLI and app update through separate channels, so the artifacts on disk drift. Proven on this machine: two CLIs at 0.18.6 / 0.18.5, and an app executable built Jun 2 — three different points in time, three potentially different cores.
The litmus test. "Where is the minutes-core executable installed?" — trick question: there isn't one. It is library code that only exists as source (crates/core/) and as .rlib build scraps, baked into the two binaries at link time. You will never see minutes-core in Activity Monitor.
